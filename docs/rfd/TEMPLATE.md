## Summary

State the proposed decision and its user-visible consequence in a short
paragraph. Do not begin with syntax.

## Problem

Describe concrete use cases, the current limitation, and why existing Spock
cannot address it. Link the language-problem issue and any WG final report.

## Goals

- Goal.

## Non-goals

- Explicitly excluded problem or tempting adjacent design.

## Design-principle fit and tensions

Identify the principles in
[`docs/governance/design-principles.md`](../governance/design-principles.md)
that govern this decision. Explain how the proposal fits them and where it
creates a genuine tension. A proposal to revise a principle must say so
explicitly rather than treating the change as incidental.

## Background and evidence

Summarize observations, user cases, implementation evidence, and relevant
constraints. Distinguish measured facts from hypotheses.

## Prior art

Explain how comparable languages or systems address the problem and which
parts do or do not transfer to Spock. Link primary sources where possible.

## Proposed design

Specify the complete design. Define terminology, the public surface, and how
it composes with existing Spock features.

### Semantics and invariants

State behavior precisely enough that the specification and conformance tests
could be written without guessing. Include invalid programs, diagnostics or
runtime failures, evaluation rules, and invariants where applicable.

### Examples

Include realistic positive examples, boundary cases, and counterexamples.
Examples in an RFD are explanatory and remain non-normative.

## Alternatives considered

Include credible alternatives, their strongest case, and why they were not
chosen. Include “make no language change” as an alternative.

## Compatibility and migration

Describe source, runtime, protocol, data, and tooling compatibility. Explain
edition/version gating, migration, rollout, and rollback where applicable.

## Consequences

Address security, privacy, performance, operability, implementation complexity,
teachability, and long-term evolution as applicable. State why an omitted
dimension is not applicable.

## Implementation and conformance plan

List implementation components, sequencing, tracking issues, required
conformance tests, documentation work, and ownership. Acceptance does not
promise that this plan will be scheduled.

## Specification changes

Name the exact files and sections under `docs/spec/` that would change. Sketch
the normative changes without treating this RFD as the specification.

## Unresolved questions

- Question that must be resolved before review or decision.
